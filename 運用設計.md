# 運用設計.md — GitHub × Supabase × アプリUI の全体設計

> E.L.Pの組織情報基盤の設計書。「知識はGitHub、データはSupabase、入力はUI、AIは両方を読む」が基本原則。

## 1. 全体アーキテクチャ

```
        従業員（スマホ/PC）                松尾・Claude Code
              │                                │
              ▼                                ▼
   ┌──────────────────┐   読み書き   ┌──────────────────┐
   │  E.L.P アプリ(UI)  │◄──────────►│   Supabase (DB)    │
   │  Next.js/Vercel    │             │  タスク/日報/営業成績│
   │  ・日報提出         │             │  /議事録/通話ログ    │
   │  ・タスク管理        │             └────────┬─────────┘
   │  ・成績ダッシュボード │                      │ 日次/週次で
   └──────────────────┘                      │ Markdownダイジェスト自動生成
                                              ▼
   イノベラ ──要約──► Supabase      ┌──────────────────┐
        └──LINE通知(東さん)         │  GitHub (e.l.p-)   │
                                   │  仕様書・マニュアル   │
                                   │  KPI定義・ルーティング│
                                   └──────────────────┘
```

## 2. 役割分担

| | GitHub（このリポジトリ） | Supabase |
|---|---|---|
| 入るもの | E.L.P.md、マニュアル、KPIの**定義**、イノベラ仕様、CLAUDE.md | タスク、日報、営業成績の**数値**、議事録、通話ログ、従業員・顧客マスタ |
| 性質 | 低頻度更新・長文・AIが文脈として読む | 高頻度更新・構造化・集計クエリする |
| 書き手 | 松尾＋Claude Code | 従業員がUIから、自動化パイプラインがAPIから |

**判断基準**: 「集計したくなるか」。集計したい（提出率・件数・金額）→ DB。やり方・定義・経緯を読みたい → GitHub。

## 3. DBテーブル構成

| テーブル | 内容 | 元のmd |
|----------|------|--------|
| `employees` | 従業員マスタ（名前、役割、LINE ID） | 従業員.md の実体 |
| `customers` | 顧客マスタ（名前、取引状況、注意事項） | 顧客一覧.md の実体 |
| `tasks` | タスク（担当、期限、状態、発生元リンク） | タスク.md を置き換え |
| `daily_reports` | 日報（従業員、日付、本文、数値） | 新規 |
| `sales_records` | 営業成績（従業員、顧客、月、件数/金額） | 新規 |
| `meeting_notes` | 議事録（日付、相手、本文MD、決定事項） | 議事録/ と併用 |
| `call_logs` | イノベラ通話要約（uniqid、要約、緊急度） | 通話ログ/ を置き換え |

マスタ移行後、従業員.md・顧客一覧.mdは「マスタはDB。ここには運用ルールのみ」と書き換える。

## 4. GitHub連動 — 一方向同期のみ（双方向同期は作らない）

1. **DB → GitHub（自動・日次/週次）**: n8n または Vercel Cron がダイジェストMarkdownを自動コミット
   - `日報/2026-W24.md`（週次まとめ）、`営業成績/2026-06.md`（月次サマリ）など
   - 目的: git履歴による監査、人間の読み物、AIの文脈
2. **リアルタイム参照はDB直接**: Claude CodeはSupabase MCP経由でSQLを実行できる。「今週の日報まとめて」はダイジェストを待たずDBに聞く

## 5. アプリUI（MVP）

- **日報提出** — スマホ最適化フォーム（現場従業員がメインユーザー。最優先）
- **タスク一覧** — 自分のタスク・完了チェック・全体一覧
- **成績ダッシュボード** — 月次件数・推移（KPI.mdの定義と対応）
- 認証: Supabase Auth（メール、将来LINEログイン拡張）
- 権限: RLS で「自分の日報は自分だけ編集可」「マスタ編集は管理者のみ」

**技術スタック**: Next.js (App Router) + Supabase + Vercel。UI母体はローカルの `elp-goals`（目標・振り返りのプロトタイプ）を拡張。

## 7. 原則（破ると壊れるもの）

1. 双方向同期は作らない（DB→GitHubの一方向のみ）
2. シークレットはローカル `.env` / Doppler。リポジトリ・DBに平文で置かない
3. マスタの正は1箇所（従業員・顧客はDB。mdには書かない）
4. ダイジェスト生成は自動化する（人が手で転記する運用にしない）
